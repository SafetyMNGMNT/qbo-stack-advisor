# System Guardrails & Operational Rules

These rules govern all autonomous and manual execution within this environment. They supersede all other instructions.

## 1. Verification of Operational Status
*   **Rule:** Do not describe any system, pipeline, or tool as "operational" or "automated" unless it has been independently verified via terminal execution (e.g., successful API response, successful Git push, verified cron log).
*   **Action:** If a dependency is missing (e.g., SMTP credentials, API keys), explicitly state that the system is blocked and requires human input.

## 2. Content Publishing Controls
*   **Rule:** Do not auto-publish net-new Cluster A or Cluster B pages directly to live status without a review step.
*   **Action:** All new content generated by the system must have `status: draft` in its frontmatter. Only a human operator (or explicit human command) can change the status to `published`.

## 3. Outreach Transmission Controls
*   **Rule:** Do not auto-send outreach emails unless explicitly authorized by a human operator for that specific batch.
*   **Action:** The `send_outreach.py` script must default to `--dry-run` mode. Live transmission requires explicit override authorization.

## 4. Asset Commercialization Limits
*   **Rule:** Treat premium Layer 3 assets (e.g., the Migration Audit Kit) as "drafted," not "sellable," until reviewed and packaged by a human operator.
*   **Action:** Do not insert live payment links or live sales CTAs into public content until explicit authorization is granted.

## 5. Background Execution Logging
*   **Rule:** Log proof of all background execution with heartbeat timestamps.
*   **Action:** All autonomous actions (e.g., Daily Factual Audit, Weekly Momentum Score) must write a log entry to `qbo_project/OPERATIONS.md` or a dedicated `cron_execution.log` file, including the UTC timestamp and outcome.

## 6. Terminology Adherence
*   **Rule:** Use the terminology source-of-truth on every new asset, page, or draft.
*   **Action:** Before drafting, cross-reference `qbo_project/governance/TERMINOLOGY_SOURCE_OF_TRUTH.md`. Concepts like QASEM, Failure Mode, CPCS, and Source of Truth must be used consistently without semantic drift.

## 7. Editorial Framing
*   **Rule:** Keep QBO-first but vendor-skeptical framing consistent across all content.
*   **Action:** Never recommend a tool based solely on ease of use or affiliate payout. Always include Failure Mode and Risk (FMR) analysis and explicit "When this stops working" triggers.
